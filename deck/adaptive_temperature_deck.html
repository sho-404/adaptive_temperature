01Research log

Adaptive
Temperature

When should a model play it safe, and when should it gamble? Teaching it to decide for itself.

Model · Ministral-3-3B-Instruct  |  Task · GSM8K math  |  Method · Latent Preference Optimization
Inspired by · “Adaptive Decoding via Latent Preference Optimization” (Dhuliawala et al., 2024)
02The problem

One temperature for everything is a compromise.

Temperature is the model’s risk dial. Low is precise but rigid — great for math. High is inventive but unreliable — great for stories. Today we pick one fixed value and live with it on every prompt.

Low temperature
Plays it safe
Accurate, repeatable, but never explores. Best for facts and arithmetic.
High temperature
Takes risks
Creative and diverse, but wanders off. Best for open-ended writing.

The question: can the model choose the right setting per problem, instead of us guessing once?

03The inspiration

The paper’s idea: learn the temperature.

Bolt a small network onto a frozen model that predicts the temperature to use — trained from preference pairs (this answer beat that one). Two flavors:

Sequence-level
One temperature per answer
Decide once, up front, before generating. Simple — where we started.
Token-level
A new temperature per word
Switch settings mid-answer — e.g. safe on a constraint, bold on prose.

05Attempt 1

Predict the temperature from the question.

Read the model’s internal state at the prompt — before it writes anything — and guess the right temperature. We fed the scorer four signals:

S1
Uncertainty
How unsure it is of its next word.
S2
Top-pick margin
How clear its favorite choice is.
S3
Hidden state
Its full 3,072-number “state of mind.”
S4
Attention spread
How focused vs. scattered it is.
06The wall

It couldn’t beat a dumb rule.

The data has a quirk: easy problems want greedy, hard ones want hotter. So “always pick the lower temperature” already scores 86.7% using nothing. Our model landed exactly there.

86.7%
trivial rule — zero features
→
86.7%
our trained scorer — no better

It quietly collapsed into “always be greedy.” No real learning.

07Why it failed

The question doesn’t reveal the difficulty.

We probed directly: can anything tell an easy problem from a hard one, just from the prompt? Barely.

0.59
easy-vs-hard separability from the prompt (0.50 = coin flip)

For math, a problem’s difficulty only becomes clear while you solve it — not by looking at the question. We were reading the model at the one moment it had nothing to say.

08The pivot

Stop reading the question.
Read the reasoning.

New idea: don’t guess up front. Let the model actually work the problem, then look at its internal state after it has reasoned. Does it know, internally, whether it got the answer right?

Clean test: a problem’s correct and incorrect attempts share the identical prompt — so prompt features can’t tell them apart. Only the reasoning can.

09The finding

The model knows when it’s right.

Predicting correctness from the model’s state after reasoning — at a fixed temperature, so it isn’t a trick:

0.50
from the prompt — knows nothing
→
~0.80
from the finished reasoning — knows a lot

The signal was there all along — just at the end of the work, not the start.

10When does it know?

Late — not mid-stream.

How well it predicts its own correctness, measured at points through the reasoning:

Mid-solution the signal is weak — too late to rescue that answer by nudging the temperature. The strong signal only lands once the answer is basically written.

11The new plan

The Judge.

Don’t pick a temperature up front. Generate a few attempts, then let a learned Judge keep the best one.

Generate
A few attempts across temperatures (variety)
→
The Judge
Scores each finished answer: “how likely is this right?”
→
Keep
The highest-scored answer ✓

Temperature’s job flips: no longer the thing we predict, just the recipe for variety. The Judge — a yes/no “is this correct?” detector trained on the reasoning state we already extracted — does the deciding.

12Where we are

What’s done, what’s next.

  • Pipeline built — 4,196 pairs, features extracted
  • Per-sequence approach tested → ruled out (0.59)
  • Reasoning features extracted on the A100 (8,392 traces)
  • Confirmed: the model knows its own correctness (~0.80)
  • Train the Judge — local, minutes, data in hand
  • Test: Judge-picks-best vs. any fixed temperature
  • If it wins → confirm on held-out test set

The pivot in one line: not “which temperature?” but “which finished answer?”
